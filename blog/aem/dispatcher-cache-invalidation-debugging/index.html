Understanding AEM Dispatcher Cache

Before diving into the incident, let's review how AEM Dispatcher cache invalidation works‚Äîbecause the bug exploited a subtle interaction between two mechanisms.

The .stat File Mechanism

Dispatcher uses .stat files as invalidation markers. When content is published:

  1. AEM publish triggers a flush request to Dispatcher
  2. Dispatcher "touches" the .stat file (updates its timestamp)
  3. On next request, Dispatcher compares cached file's mtime vs .stat mtime
  4. If cached file is older than .stat, it's considered stale

The statfileslevel Configuration

The statfileslevel setting controls which directories get their own .stat file:

# dispatcher.any excerpt
/cache {
    /docroot "/var/www/html"
    /statfileslevel "2"
    
    # With level 2, .stat files exist at:
    # /var/www/html/.stat
    # /var/www/html/content/.stat  
    # /var/www/html/content/site/.stat
    # But NOT deeper (content/site/products/.stat)
}

The Incident: What Went Wrong

Initial Detection

Monitoring caught the issue via price mismatch alerts. Our synthetic monitoring was comparing rendered prices against the PIM (Product Information Management) system:

Scope Assessment

Quick analysis revealed the pattern:

  • Affected: 23% of product pages (8,247 of 35,856 products)
  • Pattern: All affected products were in specific categories
  • Timing: Products published after 11 PM Wednesday showed correct prices

‚ö†Ô∏è Critical Pattern Identified

Affected products were all within /content/site/us/en/products/electronics/tv and /content/site/us/en/products/electronics/audio. Other categories were fine.

üîç The Bug Revealed

Cached files had timestamps of 22:30, but .stat files showed 22:15. The cache was re-populated AFTER the invalidation, before new prices were fully published. Dispatcher saw newer cache files and considered them valid!

Why Only Some Pages?

The race condition affected pages that were:

  1. Flushed in the first batch (22:15)
  2. Re-cached by traffic before update completed (22:16-22:30)
  3. In paths covered by the same .stat file (statfileslevel issue)
  4. NOT re-flushed because the bulk job used path-based invalidation

The Fix: Multi-Pronged Approach

Immediate Fix (Production - 47 minutes)

We couldn't wait for a proper fix. Immediate action:

# Nuclear option: Force full cache clear for affected paths
# Run on all Dispatcher instances

curl -X POST "http://localhost/dispatcher/invalidate.cache" \
    -H "CQ-Action: Delete" \
    -H "CQ-Handle: /content/site/us/en/products/electronics/tv" \
    -H "CQ-Path: /content/site/us/en/products/electronics/tv"

curl -X POST "http://localhost/dispatcher/invalidate.cache" \
    -H "CQ-Action: Delete" \
    -H "CQ-Handle: /content/site/us/en/products/electronics/audio" \
    -H "CQ-Path: /content/site/us/en/products/electronics/audio"

# Verify cache cleared
ls /var/www/html/content/site/us/en/products/electronics/tv/
# Should be empty or minimal

Configuration Fix (Deployed same day)

1. Increase statfileslevel

# Before: statfileslevel "2"
# After: statfileslevel "4"

/cache {
    /docroot "/var/www/html"
    /statfileslevel "4"
    
    # Now .stat files at:
    # /var/www/html/content/site/us/en/.stat
    # This means each locale has independent invalidation
}

2. Add gracePeriod

/cache {
    # Grace period: treat cache as stale for N seconds after .stat touch
    # This prevents the race condition window
    /gracePeriod "300"
    
    # For 5 minutes after invalidation, ALL requests go to publish
    # Even if cached file appears "newer"
}

3. Implement Resource-Based Invalidation

/cache {
    /invalidate {
        /0000 { /glob "*" /type "deny" }
        /0001 { /glob "*.html" /type "allow" }
    }
    
    /invalidateHandler "/bin/custom-invalidate"
    
    # Custom invalidation handler deletes specific files
    # instead of relying solely on .stat mechanism
}

Application Fix (Deployed next sprint)

We implemented a custom Sling servlet to handle bulk invalidations atomically:

@Component(service = Servlet.class)
@SlingServletPaths("/bin/bulk-price-invalidate")
public class BulkPriceInvalidationServlet extends SlingAllMethodsServlet {
    
    @Reference
    private Replicator replicator;
    
    @Reference
    private ResourceResolverFactory resolverFactory;
    
    @Override
    protected void doPost(SlingHttpServletRequest request, 
                          SlingHttpServletResponse response) {
        
        String[] paths = request.getParameterValues("path");
        
        try (ResourceResolver resolver = getServiceResolver()) {
            // Step 1: Pause Dispatcher flush agents
            pauseFlushAgents();
            
            // Step 2: Complete ALL content updates
            for (String path : paths) {
                updatePriceContent(resolver, path);
            }
            resolver.commit();
            
            // Step 3: Wait for replication queue to clear
            waitForReplicationQueue();
            
            // Step 4: Single atomic flush
            for (String path : paths) {
                replicator.replicate(
                    resolver.adaptTo(Session.class),
                    ReplicationActionType.DELETE,  // Force delete, not .stat
                    path
                );
            }
            
            // Step 5: Resume flush agents
            resumeFlushAgents();
            
            response.getWriter().write("Bulk invalidation complete");
            
        } catch (Exception e) {
            log.error("Bulk invalidation failed", e);
            response.setStatus(500);
        }
    }
    
    private void waitForReplicationQueue() throws InterruptedException {
        // Poll replication queue until empty
        while (getQueueLength() > 0) {
            Thread.sleep(1000);
        }
        // Additional safety buffer
        Thread.sleep(5000);
    }
}

Dispatcher Cache Configuration Best Practices

Based on this incident and our experience across dozens of AEM implementations:

Setting Recommendation Why
statfileslevel 4-5 for large sites Granular invalidation, prevents cascade
gracePeriod 60-300 seconds Prevents race conditions during bulk updates
Flush Agent Type Resource-based, not path-based More precise invalidation
Bulk Updates Atomic with queue drain Ensures content + cache sync
Monitoring Cache freshness + .stat sync Early detection of drift

Lessons Learned

1. statfileslevel Is Critical for Large Sites

The default level of 0-2 is insufficient for sites with deep content hierarchies. Every level adds granularity but also .stat file overhead. Test your specific content structure.

2. Bulk Operations Need Special Handling

Standard page-by-page invalidation doesn't work for bulk updates. The publish-flush-cache cycle must be atomic across the entire batch.

3. High Traffic Amplifies Race Conditions

This bug existed for months but never surfaced‚Äîtraffic was low enough that caches weren't re-populated during the race window. Black Friday traffic exposed the latent bug.

4. Monitor Cache Freshness, Not Just Availability

Standard uptime monitoring (is the page loading?) didn't catch this. You need content-aware monitoring that validates the served content matches expected state.
